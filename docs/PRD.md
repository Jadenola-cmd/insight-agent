## 一、产品定位

### 核心价值主张
不是"让业务人员自助分析"，而是**帮助商业/经营分析师用AI重构自己的分析工作流**——把重复性的清洗、标准化分析、报告撰写自动化，把时间留给真正需要判断力的部分。

### 目标用户
**商业分析师 / 经营分析师**
- 有完整的分析能力和业务理解
- 痛点在于：60-80%时间消耗在数据清洗和标准化分析上
- 真正需要判断力的部分（问题定义、业务解读、策略建议）时间严重不足
- 现有工具要么需要写代码（门槛高），要么是黑箱（不可信）

### AI工具能做的
- 问题澄清与分析目标结构化
- 数据口径诊断与清洗（透明可见、人工确认）
- 标准化分析模块执行（漏斗/趋势/分群/归因）
- 初稿洞察文字生成

### AI工具做不到的（分析师的核心价值）
- 定义什么业务问题值得分析
- 判断结论的战略意义和优先级
- 理解公司政治和决策上下文
- 把数据结论转化为可执行的业务动作
- 在会议室里说服人

### 对简历/面试的定位表述
> "识别到商业分析中80%的时间消耗在数据清洗和标准化分析上，
> 设计并开发了AI分析协作工具，将从数据到初稿结论的时间从半天
> 压缩至15分钟，让分析师把时间留在真正需要判断力的环节。"

## 二、产品设计原则

1. **问题驱动，不是数据驱动**：先定义业务问题，再上传数据，AI针对性分析
2. **AI提议，人确认，再执行**：每个关键步骤给用户"看到+决定"的机会，消除黑箱
3. **透明可信**：清洗日志逐条展示，结论附置信度，让分析师知道AI做了什么、结论可不可信
4. **报告是动态文档**：支持追问，结论追加进报告，最终导出包含完整分析过程的PDF
5. **分析师是主角，AI是工具**：产品不替代分析判断，只加速执行过程

## 三、完整用户流程

```
Step 0：AI问题澄清（对话式，2-3个问题）
           ↓
Step 1：确认分析目标 + 所需数据清单
           ↓
Step 2：上传数据（支持多文件）
           ↓
Step 3：数据诊断 + 口径确认（字段含义可编辑）
           ↓
Step 4：清洗计划预览 + 用户确认执行
           ↓
Step 5：针对性分析（只跑与目标相关的模块）
           ↓
Step 6：报告输出（含置信度标注）
           ↓
Step 7：追问对话（可触发补传数据 / 结论追加进报告）
           ↓
Step 8：导出完整PDF
```

## 四、各步骤详细设计

### Step 0：AI问题澄清

对话式引导，不超过3个问题，把模糊需求转化为可分析的业务问题。

示例：
```
AI：你现在最关心的是什么？
    A. 用户行为（他们在看什么、点什么）
    B. 转化效率（哪个环节流失最多）
    C. 用户差异（不同类型用户表现如何）
    D. 趋势变化（最近数据有没有异动）

用户选B

AI：你希望分析结果用来支持什么决定？
    A. 优化页面布局
    B. 制定运营策略
    C. 向管理层汇报

用户选B，AI输出确认：
    分析目标：找出钱包页信贷产品的转化流失环节，
             为运营策略制定提供数据依据
    需要的数据：用户行为埋点表 + 转化漏斗数据
    将运行的分析模块：漏斗分析、用户分群、归因分析
```

### Step 3：数据诊断 + 口径确认

- AI推断每个字段的业务含义，用户可直接编辑修改
- 高亮显示疑似问题字段（空值率过高、疑似重复列、命名不规范）
- 多表场景：推断表间关联键，用户确认join逻辑
- 用户确认后才进入清洗阶段

### Step 4：清洗计划预览

执行前逐条展示将要执行的操作：
```
待执行操作（共5步）：
○ 重命名：Order Date → order_date
○ 删除列：amount_usd（与amount数值完全一致）
○ 填充缺失值：risk_tier 空值 → "未知"（预计影响1,024行）
○ 删除列：notes（空值率90%）
○ 去重：按user_id + timestamp（预计删除234行）
```

用户确认后执行，执行完展示清洗结果摘要。

### Step 6：报告输出

**报告结构：**
- 分析目标回顾（直接回应Step0定义的问题）
- 关键指标概览
- 各分析模块结论（三段式：结论 / 数据支撑 / 运营建议）
- 每个结论块附置信度标注

**置信度判断规则（确定性规则，不经过LLM）：**

| 维度 | 高 | 中 | 低（加警示） |
|------|----|----|------------|
| 样本量 | >1000 | 200-1000 | <200 |
| 相关字段空值率 | <5% | 5-20% | >20% |
| 分析方法 | 描述性统计/漏斗 | 回归归因（附R²） | 预测模型 |

三个维度取最低值，原因对用户可见。

### Step 7：追问对话

**现有数据能回答：**
直接基于cleaned_data分析，结论追加到报告对应章节。

**需要新数据：**
```
AI提示：回答这个问题需要借款记录表
        建议补充字段：user_id / loan_date / amount
        上传后我可以关联现有数据继续分析
```
补传后走短路径：新表诊断 → 关联 → 针对性分析 → 追加报告。

**报告动态更新示例：**
```
转化漏斗分析
├── 主分析：申请页流失率47%是瓶颈
└── 补充分析（追问）：高意向未转化用户风险等级分布
    结论：低风险用户占73%，流失主因是体验而非风险拦截
```

## 五、技术架构概要

```
LangGraph Orchestrator
    ↓
Node1: Data Diagnosis（LLM + Pandas）
    ↓
Node2: Human Confirmation（SSE暂停）
    ↓
Node3: Deterministic Transform Engine
       （LLM输出JSON plan → 固定函数执行）
    ↓
Node4: Modular Analysis Engine
       ├─ TrendInsightModule
       ├─ ComparisonModule
       ├─ SegmentationModule
       ├─ AttributionModule（预留empirical-agent统计接口）
       └─ PredictionModule（后期）
    ↓
Node5: Narrative Generator + PDF
       （三段式洞察 → HTML模板 → WeasyPrint）
```

详见 docs/ARCHITECTURE.md

## 六、竞品对比

> 待竞品测试完成后填写（测试计划：Claude.ai / Julius.ai / WorkBuddy / Kimi）

**测试维度：**
- 多表处理能力
- 口径问题识别
- 问题理解深度
- 分析结论质量
- 置信度透明度
- 追问能力
- 报告导出

**初步判断的差异化方向（测试后验证）：**
- 问题前置（Step0）：市面产品普遍从数据出发，不问业务问题
- 确定性执行：固定函数替代LLM生成代码，可靠性更高
- 口径诊断透明化：主动发现并让用户确认，不黑箱处理
- 置信度标注：几乎没有产品做这一点

## 八、与empirical-agent的关系

| 维度 | empirical-agent | InsightAgent |
|------|----------------|--------------|
| 用户 | 经济学/管理学学生 | 商业/经营分析师 |
