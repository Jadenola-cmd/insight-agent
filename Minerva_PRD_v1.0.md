# Minerva — 产品需求文档 v1.0

> 分析思维的对话伙伴
> 面向商业分析师，不是业务侧自助分析平台

---

## 一、产品定位

### 核心定义

Minerva 是一个**假设驱动的分析对话工具**。

它不是"把数据扔进去、报告吐出来"的分析流水线，而是一个陪分析师把问题想清楚、把假设排查到底的思维伙伴。

### 一句话描述

> 所有竞品从数据开始。Minerva 从问题开始。

### 产品哲学

分析的本质是假设验证，不是数据处理。

分析师真正的困境不在于"不会跑数据"，而在于：
- 问题定义模糊，不知道真正要回答的是什么
- 假设穷举不完整，漏掉关键解释角度
- 分析卡住时，不知道下一步该往哪里走
- 遇到模糊结论（如"意愿问题"），缺乏追问的动力和路径

Minerva 解决的是这些困境，不是数据处理的效率问题。

---

## 二、目标用户

### 核心用户画像

**有分析能力、经常卡在"分析到一半"的商业/数据分析师**

具体特征：
- 能独立拆解问题到一两层，但容易在"全维度无区分度"时失去方向
- 熟悉 SQL 和数据处理，执行不是瓶颈，思维深度是瓶颈
- 工作场景中缺乏"会追问的对话伙伴"——业务侧不懂分析，分析侧没有时间深聊
- 分析结论经常停留在"意愿问题""外部环境"等模糊归因，无法推动业务侧采取行动

### 不是目标用户

- 业务侧想自助拿结论的非技术用户
- 需要自动化报表的运营人员
- 学术研究场景（严格统计检验优先于业务洞察）

## 三、核心交互：三阶段对话

整个分析过程是一次持续对话，分三个阶段推进。AI 在每个阶段扮演不同角色。

### 阶段一：问题定义（苏格拉底模式）

**AI 角色：追问者**

用户描述观察到的现象，AI 不急于给方向，而是先追问，帮用户把模糊的观察变成清晰、有边界的问题陈述。

追问聚焦三个方向：
1. **基准问题**：这个指标是多少才算正常？下滑之前的基准是什么？
2. **定义问题**：这个指标背后真正代表的业务现实是什么？
3. **范围问题**：这是一个独立问题，还是某个更深层问题的症状？

追问轮数：≤3轮，第3轮强制收敛输出问题陈述。

**阶段输出：问题陈述卡片**
```
问题：[经过校验的问题描述]
指标基准：[正常范围 / 历史水位]
业务含义：[这个指标真正代表什么]
分析目标：[我们真正想回答的问题]
```

---

### 阶段二：假设树构建（咨询顾问模式）

**AI 角色：结构化思维伙伴**

基于问题陈述，AI 与用户共同穷举可能的原因假设，结构化成树状，标注优先级。

AI 的核心价值在于：
- 补充用户没有想到的假设角度
- 识别假设之间的逻辑关系（并列/递进/互斥）
- 基于业务常识判断哪个假设优先验证（验证成本低 + 解释力强）
- 把用户提到但没有重视的细节（如"偶尔莫名变高"）转化为可验证假设

**阶段输出：假设树**
```
问题：支用率持续下滑

├── 【供给侧】产品或流程发生了变化
│   ├── 假设1.1 授信策略收紧，拿到额度的人本来需求就更低
│   └── 假设1.2 产品体验某个环节出现问题导致流失
│
├── 【需求侧】用户借款意愿真实下降
│   ├── 假设2.1 季节性需求波动（节假日/发薪日规律）⭐优先验证
│   ├── 假设2.2 竞品分流（同期其他平台低息活动）
│   └── 假设2.3 宏观消费意愿收缩
│
└── 【数据侧】指标统计口径问题
    └── 假设3.1 授信到支用的时间间隔拉长（未支用≠不想支用）⭐优先验证
```

每个假设节点包含：假设描述、优先级（⭐标注）、验证所需数据、预期验证方式。

---

### 阶段三：验证执行（数据分析师模式）

**AI 角色：确定性执行者**

用户与 AI 商定好验证哪个假设，AI 自动执行数据分析，返回结果，结果回到对话里继续讨论，更新假设树状态。

执行原则（继承自现有架构）：
- LLM 只负责生成分析计划（JSON 格式），不生成可执行代码
- 所有数据操作通过固定函数集合执行，保证确定性
- 每次验证结果标注置信度（三维度：样本量/数据质量/方法适配性）

**验证结果回到对话**：
```
假设2.1 验证结果：季节性规律存在，但解释力有限
- 节假日期间支用率确实高于均值 23%
- 但非节假日期间的下滑趋势仍然显著
- 结论：季节性是部分原因，不是主因
→ 建议：继续验证假设3.1（时间间隔变化）
```

假设树对应节点状态更新为：✓ 部分验证 / ✗ 排除 / ? 待验证

## 四、分析地图（侧边栏）

分析地图是假设树的可视化实时呈现，贯穿整个对话过程，显示在界面侧边栏。

**核心功能**：
- 让用户随时知道"我现在在哪个阶段"
- 记录哪些假设已验证、哪些排除、哪些待验证
- 防止长对话中迷失方向

**节点状态**：
- ⬜ 待验证
- 🔄 验证中
- ✅ 已验证（支持）
- ❌ 已排除
- ⚠️ 部分验证

**三个阶段在地图上的体现**：
```
[问题定义] ✅ 已完成
  └─ 支用率下滑：从11%降至5%，持续3个月

[假设树] 🔄 验证中（2/6）
  ├─ 假设1.1 授信策略收紧 ⬜
  ├─ 假设1.2 产品体验流失 ⬜
  ├─ 假设2.1 季节性规律 ✅ 部分
  ├─ 假设2.2 竞品分流 ⬜
  ├─ 假设2.3 宏观收缩 ⬜
  └─ 假设3.1 时间间隔拉长 🔄

[综合结论] ⬜ 待生成
```

## 五、技术策略

### 模型选择

前两阶段（问题定义 + 假设树构建）强依赖模型质量，是产品差异化的核心。

**只接入顶级模型**，当前选择：
- 主力：DashScope DeepSeek（成本可控）
- 备用：Claude（复杂追问场景）

这不是成本约束，是产品定位：**用最强的模型做最难的部分——帮你想清楚问题。**

### 执行引擎（复用现有架构）

第三阶段的数据验证执行，直接复用现有 LangGraph 后端：
- 清洗引擎（node3_transform）
- 分析模块（Trend / Comparison / Segmentation / Attribution）
- 置信度标注（三维度规则计算）
- HTML 报告生成

**不推翻已有工作，接入新的对话前端。**

### 对话记忆

AI 在整个分析过程中持续引用用户已经说过的关键信息：

- 问题陈述卡片（阶段一输出）全程保留在上下文
- 假设树状态实时同步到每次 LLM 调用的 prompt
- 用户提到的业务背景细节（如"偶尔莫名变高"）被标记并在后续追问中引用

## 六、竞品差异化

| 维度 | Julius / DeepSeek / Kimi | Minerva |
|------|--------------------------|---------|
| 起点 | 用户扔数据 | 用户描述业务困惑 |
| AI 角色 | 数据处理执行者 | 分析思维对话伙伴 |
| 核心价值 | 帮你跑完分析 | 帮你想清楚该怎么分析 |
| 问题定义 | 跳过，默认已定义 | 第一步，追问校验 |
| 假设管理 | 无 | 假设树，贯穿全程 |
| 用户感受 | "AI 帮我做完了" | "我通过 AI 想明白了" |
| 适合用户 | 需要快速出结论的人 | 需要想清楚问题的分析师 |

**核心差异只有一个，但很深**：所有竞品假设问题已经定义好了，Minerva 从问题定义开始。

## 七、用户验证计划

### 第一阶段：深度共创（上线前）

- 目标：2-3 名有真实分析工作的分析师
- 方式：用真实工作案例走完整三阶段流程
- 核心问题：在哪个节点 AI 的追问真正推动了你的思考？
- 成功标准：用户说出"这个角度我之前没想到"

### 第二阶段：小圈子推广

- 渠道：分析师社群、小红书（分析师向内容）
- 定位：不是效率工具，是思维工具
- 核心内容：真实分析案例的完整推演过程
